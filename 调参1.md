结论：**现在不建议继续把主要精力投在 V2 调参上，应该转向强化学习信号控制。**

你们已经不是“V2 只调了一两次还不确定”的状态，而是已经做过一轮比较完整的中等规模探索：`entity_output_bias`、temporal pooling、graph residual、V2 专属超参、phase state、phase embedding、graph attention bias、queue weighted loss、单组件消融都试过了，但 V2 仍没有稳定超过 V1。文档里也明确总结：当前不建议继续小修小补 V2，正式展示和论文主线优先采用 V1 或稳定模型，V2 作为探索性扩展保留。

---

## 我的判断：V2 到此为止，除非做结构性重构

从结果看，V2 不是完全没价值，但它的价值不足以支撑继续投入。

目前最关键的对比是：

| 方案                   | V1 MAE | V2 MAE | 判断            |
| -------------------- | -----: | -----: | ------------- |
| 初始 full_v3           | 0.8545 | 0.8591 | V2 未超过 V1     |
| V2 tuned             | 0.8545 | 0.8610 | V2 仍未超过 V1    |
| phase_state_v1       | 0.8727 | 0.8714 | V2 仅在弱口径内略优   |
| phase_embed_graph_v1 | 0.8434 | 0.8946 | V1 明显更强，V2 变差 |
| V2 ablation 最佳       | 0.8545 | 0.8726 | V2 仍明显落后      |

这里最重要的是：**phase embedding + graph 信息让 V1 变强，却让 V2 变差**。这说明不是“控制特征没用”，而是 V2 当前融合方式不适合你们现在的数据和任务。继续加小组件，大概率还是在 V2 内部打转。

你们当前数据规模是 `X=[2448, 12, 1512]`、`y=[2448, 15, 453]`，movement 数量 151，输出维度 453。这个规模对 V1 这种扁平 Transformer 是够用的，但对显式时空 V2 来说偏小，尤其当它还要学习 movement graph、phase embedding、空间关系和多目标输出时，容易出现“结构更高级，但训练不稳”的情况。

所以我建议：

**不要继续小修 V2。**
**不要再为了让 V2 超过 V1 耗时间。**
**把 V1/当前最稳模型定为预测模块主模型，然后转向信号控制闭环。**

---

## V2 怎么处理最合适？

建议把 V2 定位为：

> 已实现的探索性时空预测模型，用于证明团队尝试了 movement 级时空建模、相位特征融合和图关系增强；但在当前样本规模和图构造方式下，整体精度未稳定超过 V1，因此系统默认采用更稳健的 V1 预测器。

这句话对论文和答辩都很安全。它不是承认失败，而是体现你们做了对比和消融，并且选择了工程上更稳的方案。

你们可以保留 V2 的两个亮点：

1. **短期预测上有局部信号**：初始 V2 的 `MAE@5` 略优于 V1。
2. **控制扰动子集上有潜力**：tuned V2 在 `control_perturbation` 子集上优于 V1。

但不要把它作为主线模型。文档里已经说明 tuned V2 在控制扰动子集上表现更好，但 overall 没有超过 V1，这个可以作为“未来工作”而不是当前核心成果。

# 下一步：转向强化学习信号控制

你们现在已经具备做 RL 信号控制的基础：

* 已有 movement 级数据；
* 已有 151 个 movement；
* 已有 14 个信号灯路口；
* movement catalog 质量是可用的，没有重复 movement id、没有 missing phase、zone quality 全部 ok；
* 预测目标已经覆盖 `arrival_flow / mean_speed / queue_veh`，足够作为信号控制状态输入。

所以现在最有价值的工作不是再调预测，而是建立：

```text
SUMO 仿真环境 → movement 状态采集 → RL 控制动作 → 仿真反馈 → 控制效果评估
```

这才是数字孪生平台最有展示力的闭环。

---

## 强化学习信号控制的推荐抓手

### 阶段 1：先做单路口控制，不要一上来全网

建议先选一个结构完整、movement 数量标准的四进口信号灯，比如 movement 数量为 12 的路口。你们质量报告里多个 tls 都是 12 个 movement、4 个 incoming_edges、左直右齐全，例如 `12254671324`、`12254692358`、`12260384797`、`1704063278`、`1704063290`、`J42`、`J49`、`J55` 等。

先做一个路口的 RL，原因是：

* 容易稳定训练；
* 容易和固定配时对比；
* 容易做可视化；
* 容易解释 reward；
* 不会被多路口协调问题拖死。

---

## 阶段 2：动作空间不要让 RL 直接生成红绿灯字符串

动作空间建议用**合法相位切换**，不要让智能体直接输出任意红黄绿序列。

首版动作可以很简单：

```text
0 = 保持当前绿灯相位
1 = 切换到下一个合法绿灯相位
```

更稳一点：

```text
0 = 保持当前相位
1 = 切换到下一主相位
2 = 切换到下一左转相位
```

但第一版建议只做二动作：

```text
keep / switch
```

这样最不容易出错。黄灯、全红、安全间隔仍由 SUMO 信号程序或封装环境处理。

---

## 阶段 3：状态空间直接用 movement 级状态

每个控制步，比如每 10 秒或 30 秒，提取当前路口下所有 movement 的状态。

建议状态包括：

```text
queue_veh
arrival_flow
mean_speed
is_green
phase_elapsed_s
green_remaining_s
```

如果接入预测，则再加：

```text
pred_queue_5min
pred_arrival_flow_5min
pred_mean_speed_5min
```

首版建议先做两种环境：

```text
RL-no-pred：只用当前状态
RL-with-pred：当前状态 + V1 预测状态
```

这样你们可以证明预测模块不是孤立的，而是能增强控制决策。

---

## 阶段 4：reward 要简单，先别太复杂

第一版 reward 建议：

```text
reward = - (
    1.0 * total_queue_veh
  + 0.2 * total_waiting_time
  + 0.1 * total_halting_vehicles
  - 0.5 * discharged_vehicles
)
```

也可以更简单：

```text
reward = discharged_vehicles - queue_penalty
```

例如：

```python
reward = 0.5 * discharged_vehicles - 1.0 * total_queue_veh - 0.2 * total_waiting_time
```

不要一开始加碳排、舒适性、公平性、冲突风险，否则 reward 调不稳。

---

## 阶段 5：控制基线必须做

RL 一定要和基线比。建议至少做 3 个：

```text
Baseline 1：当前 SUMO/Webster 固定配时
Baseline 2：简单 actuated rule，根据排队延长绿灯
Baseline 3：DQN 控制
```

如果时间紧，至少做：

```text
fixed-time vs DQN
```

评价指标：

```text
average_delay
total_waiting_time
mean_queue_veh
max_queue_veh
throughput
mean_speed
stops
emergency_braking / collision 如果有事故推演
```

## 第 2 步：先跑单路口 DQN

先不要 PPO、SAC、MARL。
DQN 足够展示“智能控制”。

DQN 输入：

```text
state_dim = movement_count_of_tls * feature_count + phase_context
```

输出：

```text
Q(keep), Q(switch)
```

控制间隔：

```text
control_interval_s = 10 或 30
```

最小绿灯：

```text
min_green_s = 10 或 15
```

防止频繁切相位。

---

## 第 3 步：加入预测增强

在 `state_builder.py` 中加一个开关：

```json
"use_prediction": true
```

如果开启，就调用当前 V1 预测器，把未来 5 分钟的 movement 状态拼进 RL state。

对比：

```text
DQN-current-state
DQN-current+prediction
```

这会是你们论文和视频里非常有价值的一张图：

```text
固定配时 < RL 当前状态 < RL + 数字孪生预测
```

---

## 第 4 步：事故场景下验证控制

事故推演不一定要训练专门的事故控制器，先做评估即可：

```text
固定配时 + accident
RL + accident
RL + prediction + accident
```

观察：

```text
排队回溢是否减少
恢复时间是否缩短
throughput 是否提高
```

这非常适合比赛主题里的“城市安全与韧性”。
