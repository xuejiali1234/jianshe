下面是基于你们**当前仓库代码 + 上传的配置/指标文件**给出的诊断。先说结论：

**Transformer V2 现在不是“结构一定比 V1 差”，而是 V2 当前实现还没有把 movement 级建模的优势真正用起来。**
目前 V2 只是把输入拆成 `[movement, feature]` 后做“每个 movement 的时间编码 + 全连接 movement 空间编码”，但没有使用 `movement_graph.json`，没有相位/拓扑 attention bias，也没有足够的 entity-specific 输出能力；同时训练超参数沿用了 V1 的轻量设置，对 V2 不友好。所以 V2 目前只是“拆维版 Transformer”，还不是“图约束时空 Transformer”。

你们当前数据规模是 `X=[2448, 12, 1512]`、`y=[2448, 15, 453]`，151 个 movement，每个 movement 预测 3 个目标，当前最佳模型仍是 `transformer_v1`。 当前背景里给出的整体指标是：V1 `MAE=0.8545, RMSE=1.7882, WAPE=0.3173`，V2 `MAE=0.8591, RMSE=1.8218, WAPE=0.3190`，V2 只是小幅落后，但没有形成结构性优势。

# 1. V2 的输入 reshape 是否合理？

## 结论：**当前 reshape 基本合理，但非常脆弱，需要加显式校验。**

当前 dataset 的 movement 输入特征是按下面这个顺序生成的：

```python
feature_names = [
    f"{entity_id}__{feature}"
    for entity_id in entity_ids
    for feature in input_features
] + ["tod_sin", "tod_cos"]
```

也就是说，前面是 `151 × 每个 movement 特征数`，最后两个是全局时间特征 `tod_sin/tod_cos`。([GitHub][1])

当前 movement 输入特征包括：

```python
arrival_flow
discharge_flow
mean_speed
occupancy
queue_veh
queue_meter
incident_flag
phase_id
phase_elapsed_s
green_remaining_s
```

也就是带控制特征时每个 movement 10 个输入特征。代码里 `MOVEMENT_INPUT_FEATURES` 的定义也正是这个顺序。([GitHub][1])

因此在当前 full V3 数据下：

```text
input_size = 1512
time_feature_size = 2
entity_body_size = 1512 - 2 = 1510
num_entities = 151
entity_input_size = 1510 / 151 = 10
```

所以 V2 里这段 reshape 在当前数据上是成立的：

```python
entity_flat = x[:, :, : self.num_entities * self.entity_input_size]
entity_values = entity_flat.view(
    batch_size,
    history_steps,
    self.num_entities,
    self.entity_input_size,
)
```

这对应的是：

```text
[B, 12, 1510] → [B, 12, 151, 10]
```

代码中的 reshape、`entity_input_size` 推断和 `output_size == num_entities * target_size` 校验都在 `SpatioTemporalTransformerForecaster.__init__` 里实现了。([GitHub][2])

## 但这里有两个隐患

### 隐患 A：V2 默认相信 feature_names 的顺序永远正确

只要以后 dataset 构造逻辑变了，比如把 `tod_sin/tod_cos` 插到每个 entity 里面，或者 feature 顺序变成：

```text
arrival_flow_all_entities
speed_all_entities
queue_all_entities
...
```

当前 reshape 就会直接错位，但不一定报错。

### 隐患 B：`time_feature_size` 是通过最后两个 feature 名判断的

训练代码里是：

```python
time_feature_size = 2 if dataset.feature_names[-2:] == ["tod_sin", "tod_cos"] else 0
```

这在当前数据上是对的，但过于隐式。([GitHub][3])

## 建议最小修复

在 `train_torch_model()` 构建 V2 前加显式校验：

```python
def infer_v2_feature_layout(dataset):
    time_names = ["tod_sin", "tod_cos"]
    has_time = dataset.feature_names[-2:] == time_names
    time_feature_size = 2 if has_time else 0

    entity_feature_count = (len(dataset.feature_names) - time_feature_size) // len(dataset.edge_ids)
    expected_total = len(dataset.edge_ids) * entity_feature_count + time_feature_size

    if expected_total != len(dataset.feature_names):
        raise ValueError(
            f"Bad V2 feature layout: feature_names={len(dataset.feature_names)}, "
            f"entities={len(dataset.edge_ids)}, inferred entity_feature_count={entity_feature_count}, "
            f"time_feature_size={time_feature_size}"
        )

    # 强校验前几个和最后几个字段
    for entity_id in dataset.edge_ids[:3]:
        prefix_count = sum(1 for f in dataset.feature_names if f.startswith(f"{entity_id}__"))
        if prefix_count != entity_feature_count:
            raise ValueError(
                f"Entity {entity_id} has {prefix_count} features, "
                f"expected {entity_feature_count}"
            )

    return time_feature_size, entity_feature_count
```

然后 V2 config 改成：

```python
time_feature_size, entity_input_size = infer_v2_feature_layout(dataset)

model_config = {
    "input_size": input_size,
    "output_size": output_size,
    "horizon_steps": horizon_steps,
    "num_entities": len(dataset.edge_ids),
    "entity_input_size": entity_input_size,
    "target_size": len(dataset.targets),
    "time_feature_size": time_feature_size,
    ...
}
```

这不是为了提升指标，而是为了防止 V2 悄悄吃错数据。

# 2. time_feature_size / entity_input_size 是否正确？

## 结论：**当前 full-control 数据上是正确的：time_feature_size=2，entity_input_size=10。**

根据 `p4_training_summary.json`，当前完整输入是 `1512`，无控制特征消融输入是 `1059`。
结合 151 个 movement：

```text
with_control:
1512 - 2 = 1510
1510 / 151 = 10

without_control:
1059 - 2 = 1057
1057 / 151 = 7
```

这正好对应：

```text
with_control 每个 movement 10 个特征：
arrival_flow
discharge_flow
mean_speed
occupancy
queue_veh
queue_meter
incident_flag
phase_id
phase_elapsed_s
green_remaining_s

without_control 每个 movement 7 个特征：
arrival_flow
discharge_flow
mean_speed
occupancy
queue_veh
queue_meter
incident_flag
```

所以不是 `entity_input_size` 算错导致 V2 落后。

## 真正的问题是：time feature 被全体 movement 共享相加，表达力偏弱

当前 V2 是这样处理时间特征的：

```python
time_features = x[:, :, -self.time_feature_size :]
z = z + self.time_proj(time_features).unsqueeze(2)
```

这会把同一份 `tod_sin/tod_cos` 加到所有 movement 上。这个本身没错，但它只表示“当前时间段”，不能表达：

```text
这个 movement 当前相位是否马上结束
这个 movement 的相位在未来 horizon 里是否会变绿
不同 tls 的相位周期差异
```

你们已经有 `phase_id / phase_elapsed_s / green_remaining_s`，但它们现在只是普通连续数值。尤其 `phase_id` 本质是类别变量，却被当作连续数值喂给 `Linear`，这对 V2 不友好。V1 因为直接看完整 flatten 输入，反而可能通过大 head 暴力拟合相位模式。

## 建议最小修复

先不要大改数据集，只在 V2 里增加一个 `phase_id` embedding 的方向是不现实的，因为当前 V2 只拿到统一的 `entity_values`，不知道哪个 feature index 是 `phase_id`。更小的做法是：

在训练构建 V2 时传入：

```python
entity_feature_names = dataset.feature_names[:len(dataset.edge_ids) * entity_input_size]
per_entity_features = [
    name.rsplit("__", 1)[1]
    for name in entity_feature_names[:entity_input_size]
]
```

然后在 V2 内部保存 feature index：

```python
self.phase_id_index = per_entity_features.index("phase_id") if "phase_id" in per_entity_features else None
```

首轮不一定要做 phase embedding，但至少要把这些 index 存进 artifact，方便之后定位问题。

# 3. temporal encoder 和 spatial encoder 的顺序是否合理？

## 结论：**当前“先 temporal 后 spatial”不是错误，但对你们这个任务不够好。**

当前 V2 的 forward 是：

```python
# [B, T, N, F] → entity projection
z = self.entity_proj(entity_values)

# 加 time feature 和 entity embedding
z = z + time_proj(...)
z = z + entity_embedding(...)

# temporal: 每个 movement 单独过 12 步历史
z = z.permute(0, 2, 1, 3).reshape(B * N, T, D)
z = temporal_encoder(z)
z = z[:, -1, :].view(B, N, D)

# spatial: 只对最后时刻 summary 做 movement attention
z = spatial_encoder(z)
```

也就是说，**空间编码只能看到每个 movement 的最终 temporal summary，看不到完整时序里的跨 movement 交互。** 这段逻辑在仓库 `torch_models.py` 里很清楚：V2 先将 `[B,T,N,D]` 改成 `[B*N,T,D]` 做 temporal encoder，然后只取 `z[:, -1, :]`，最后才做 `spatial_encoder(z)`。([GitHub][2])

这对普通道路预测可能还能接受，但你们是**信号灯交叉口 movement 预测**，核心关系是：

```text
相同 tls 的 movement 受同一个信号周期影响
同一进口道左/直/右共享排队空间
不同相位之间互斥
上游 movement 会传递到下游 movement
```

这些关系很多是**时间同步关系**。比如当前第 8 分钟西向直行放行，可能影响第 9–10 分钟下游 movement 的 arrival_flow。如果 temporal 先把每个 movement 压成一个 final token，再做 spatial，就会丢掉“哪个时间点发生了什么交互”。

## 建议顺序

### 最小改动版：保留 temporal → spatial，但不要只用最后一步

把：

```python
z = self.temporal_encoder(z)
z = z[:, -1, :].view(batch_size, self.num_entities, -1)
```

改成：

```python
z = self.temporal_encoder(z)
z_last = z[:, -1, :]
z_mean = z.mean(dim=1)
z = 0.7 * z_last + 0.3 * z_mean
z = z.view(batch_size, self.num_entities, -1)
```

理由：交通状态不是 NLP 句子，最后一步不一定足以代表整个 12 分钟历史。均值池化能保留过去窗口的趋势。

### 更推荐版：spatial-first per time step，再 temporal

把流程改成：

```text
[B, T, N, F]
→ entity_proj
→ 对每个时间步做 spatial encoder: [B*T, N, D]
→ 再对每个 movement 做 temporal encoder: [B*N, T, D]
→ head
```

伪代码：

```python
z = self.entity_proj(entity_values)
z = z + time_proj + entity_embedding

# spatial first
z = z.reshape(batch_size * history_steps, self.num_entities, -1)
z = self.spatial_encoder(z, mask=spatial_mask)
z = z.view(batch_size, history_steps, self.num_entities, -1)

# temporal second
z = z.permute(0, 2, 1, 3).reshape(batch_size * self.num_entities, history_steps, -1)
z = self.temporal_positional(z)
z = self.temporal_encoder(z)
z = z[:, -1, :].view(batch_size, self.num_entities, -1)
```

这比当前顺序更符合交通传播：**先在同一时间片上聚合邻近 movement，再建模每个 movement 的历史变化。**

但如果你们想“最小改动”，先做 `last + mean pooling` 即可。

# 4. 当前训练超参数是否对 V2 不友好？

## 结论：**是。现在训练设置更偏向 V1，V2 没吃到足够训练预算。**

当前所有 torch 模型共用：

```python
batch_size = min(64, len(train_ds))
criterion = nn.HuberLoss()
optimizer = AdamW(lr=1e-3, weight_decay=1e-4)
patience = 4
max_epochs = 12
```

这些设置在 `training.py` 里是统一写死的。([GitHub][3])

问题在于 V2 比 V1 更难优化：

| 项目         | Transformer V1              | Transformer V2                |
| ---------- | --------------------------- | ----------------------------- |
| 输入         | `[B, T, 1512]` flatten 全局输入 | `[B, T, 151, 10]` entity 局部输入 |
| 空间建模       | 隐式靠大 input_proj/head 拟合     | 显式 temporal + spatial 两段      |
| 输出头        | 一次性输出 `15×453`，每个输出有独立权重    | 每个 entity 共享 head 输出 `15×3`   |
| 当前 d_model | 128                         | 96                            |
| 当前空间约束     | 无                           | 无，仍是全连接 attention             |

V2 的 `d_model=96`，同时 head 是共享的：

```python
self.head = nn.Sequential(
    nn.LayerNorm(d_model),
    nn.Linear(d_model, d_model),
    nn.GELU(),
    nn.Dropout(dropout),
    nn.Linear(d_model, horizon_steps * target_size),
)
```

然后对每个 entity 使用同一个 head。([GitHub][2])

这意味着 V2 的输出层远小于 V1。V1 的最后一层直接输出：

```text
horizon_steps * output_size = 15 * 453 = 6795
```

而 V2 的每个 movement head 只输出：

```text
15 * 3 = 45
```

虽然 entity embedding 给了不同 movement 一些身份信息，但最终 head 仍是共享映射。对 151 个 movement 来说，这可能是 V2 不如 V1 的最大结构性原因之一：**V1 有很强的 entity-specific 输出能力，V2 被过度共享了。**

## 建议超参数改法

V2 单独设置，不要和 V1 共用：

```python
elif kind == "transformer_v2":
    model_config = {
        ...
        "d_model": 128,
        "n_heads": 4,
        "temporal_layers": 2,
        "spatial_layers": 1,
        "dropout": 0.05,
    }

    lr = float(os.environ.get("TRAFFIC_TRAIN_LR_V2", "3e-4"))
    weight_decay = float(os.environ.get("TRAFFIC_TRAIN_WD_V2", "5e-5"))
    max_epochs = int(os.environ.get("TRAFFIC_TRAIN_EPOCHS_V2", "40"))
    patience = int(os.environ.get("TRAFFIC_TRAIN_PATIENCE_V2", "8"))
```

训练 loop 加梯度裁剪：

```python
loss.backward()
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
optimizer.step()
```

如果只允许改一组参数，我建议：

```text
d_model: 96 → 128
spatial_layers: 2 → 1
dropout: 0.1 → 0.05
lr: 1e-3 → 3e-4
epochs: 12 → 40
patience: 4 → 8
```

理由：V2 当前不是明显过拟合，而是略微欠拟合/结构优势没发挥；先降低学习率、延长训练、提高 d_model，同时减少 spatial 层数防止全连接 spatial attention 过度搅乱。

# 5. 是否应该接入 movement_graph.json 做 attention bias 或邻接聚合？

## 结论：**应该，而且这是 V2 最应该补的地方。**

你们已经有 `movement_graph.json`，里面的关系类型包括：

```text
same_tls
same_incoming_edge
same_phase
upstream_downstream
conflict_same_tls_different_phase
```

并且关系数量是：

```text
same_tls: 1546
same_incoming_edge: 284
same_phase: 1138
upstream_downstream: 42
conflict_same_tls_different_phase: 408
```

也就是说，图关系已经有了，而且刚好是 V2 应该使用的空间先验。

但当前 V2 完全没有传入图结构。`SpatioTemporalTransformerForecaster` 只定义了普通 `nn.TransformerEncoder` 作为 spatial encoder，没有 mask，也没有 attention bias。([GitHub][2])

这会导致 151 个 movement 之间做**全连接无约束 attention**。对你们这种多路口、多相位、多转向 movement 来说，很多 attention 是噪声。例如：

```text
J42 的左转 movement 没必要直接关注另一个远端交叉口的右转 movement
同一 tls / 同一进口道 / 上下游 movement 才应该优先关注
冲突相位可能应该作为特殊负相关或竞争关系处理
```

## 最小实现优先级

不要一上来写复杂 attention bias。建议先做 **邻接聚合 residual**，因为改动最小、稳定性最好。

### Step 1：从 movement_graph.json 构造邻接矩阵

新增文件：

```text
prediction/graph_utils.py
```

```python
from __future__ import annotations

import json
from pathlib import Path

import numpy as np
import torch


RELATION_WEIGHTS = {
    "same_incoming_edge": 1.0,
    "upstream_downstream": 0.8,
    "same_phase": 0.5,
    "same_tls": 0.25,
    # conflict 先不做正向聚合，避免把互斥相位混在一起
    "conflict_same_tls_different_phase": 0.0,
}


def load_movement_adjacency(
    graph_path: str | Path,
    entity_ids: list[str],
    include_self: bool = True,
) -> torch.Tensor:
    graph_path = Path(graph_path)
    payload = json.loads(graph_path.read_text(encoding="utf-8"))

    idx = {entity_id: i for i, entity_id in enumerate(entity_ids)}
    n = len(entity_ids)
    adj = np.zeros((n, n), dtype=np.float32)

    if include_self:
        np.fill_diagonal(adj, 1.0)

    adjacency = payload.get("adjacency", {})
    for src, rels in adjacency.items():
        if src not in idx:
            continue
        i = idx[src]
        for rel_type, dst_list in rels.items():
            weight = RELATION_WEIGHTS.get(rel_type, 0.0)
            if weight <= 0:
                continue
            for dst in dst_list:
                if dst in idx:
                    j = idx[dst]
                    adj[i, j] = max(adj[i, j], weight)

    # row normalize
    row_sum = adj.sum(axis=1, keepdims=True)
    row_sum[row_sum < 1e-6] = 1.0
    adj = adj / row_sum

    return torch.tensor(adj, dtype=torch.float32)
```

### Step 2：V2 加 graph residual

在 `SpatioTemporalTransformerForecaster.__init__` 里加：

```python
graph_adjacency: torch.Tensor | None = None,
graph_residual_init: float = 0.2,
```

然后：

```python
if graph_adjacency is not None:
    self.register_buffer("graph_adjacency", graph_adjacency.float())
    self.graph_gate = nn.Parameter(torch.tensor(graph_residual_init, dtype=torch.float32))
else:
    self.graph_adjacency = None
    self.graph_gate = None
```

在 spatial encoder 前加：

```python
if self.graph_adjacency is not None:
    z_graph = torch.einsum("ij,bjd->bid", self.graph_adjacency, z)
    z = z + torch.tanh(self.graph_gate) * z_graph
```

也就是：

```python
z = z[:, -1, :].view(batch_size, self.num_entities, -1)

if self.graph_adjacency is not None:
    z_graph = torch.einsum("ij,bjd->bid", self.graph_adjacency, z)
    z = z + torch.tanh(self.graph_gate) * z_graph

z = self.spatial_encoder(z)
```

这个改动比 attention bias 小很多，但已经能把 `same_tls / same_phase / same_incoming_edge / upstream_downstream` 这些结构先验注入进去。

### Step 3：训练时传入 graph

在 `training.py` 的 V2 分支里：

```python
from .graph_utils import load_movement_adjacency

graph_path = Path("configs/movement_graph.json")
graph_adjacency = None
if graph_path.exists():
    graph_adjacency = load_movement_adjacency(graph_path, dataset.edge_ids)

model_config = {
    ...
    "graph_adjacency": graph_adjacency,
}
```

如果 `movement_graph.json` 在项目里不是 configs 路径，就从 config 增加字段：

```json
"movement_graph_file": "configs/movement_graph.json"
```

你们当前 `movement_config.json` 质量报告显示 151 个 movement 全部 `zone_quality=ok`，没有重复 movement id，也没有 missing phase，这说明图对齐的基础是可靠的。

# 6. 给出最小改动方案，让 V2 更可能超过 V1

下面是我建议直接给 Codex 的最小改动方案，按优先级执行。

---

## Patch A：修复 V2 的 entity-specific 输出能力

这是最小、收益可能最大的结构修复。

### 问题

V2 当前所有 movement 共用同一个 head：

```python
self.head(z) → [B, N, H * target_size]
```

这会让不同 movement 的输出只通过 `z` 区分。V1 则直接有 `15×453` 的大输出层，天然 entity-specific。([GitHub][2])

### 修改

在 V2 里增加每个 entity 的输出 bias：

```python
self.entity_output_bias = nn.Embedding(
    num_entities,
    horizon_steps * target_size,
)
```

forward 里：

```python
raw_out = self.head(z)

entity_ids = torch.arange(self.num_entities, device=x.device)
raw_out = raw_out + self.entity_output_bias(entity_ids).view(
    1,
    self.num_entities,
    self.horizon_steps * self.target_size,
)

out = raw_out.view(
    batch_size,
    self.num_entities,
    self.horizon_steps,
    self.target_size,
)
```

这只增加：

```text
151 × 45 = 6795
```

个参数，刚好补偿 V2 对每个 movement 缺少专属输出偏置的问题，参数量很小，风险低。

---

## Patch B：V2 加 graph residual，而不是全连接裸 spatial attention

### 修改点

新增 `prediction/graph_utils.py`，在 V2 里加入：

```python
z_graph = A_norm @ z
z = z + gate * z_graph
```

推荐关系权重：

```python
same_incoming_edge: 1.0
upstream_downstream: 0.8
same_phase: 0.5
same_tls: 0.25
conflict_same_tls_different_phase: 0.0
self: 1.0
```

理由：

* 同一进口道 movement 共享排队空间，最强；
* upstream/downstream 是交通传播关系，重要但数量少；
* same_phase 表示共同放行；
* same_tls 表示同路口弱相关；
* conflict 先不做正向聚合，避免把互斥相位混成同类。

---

## Patch C：V2 pooling 从 last-only 改为 last + mean

### 当前

```python
z = self.temporal_encoder(z)
z = z[:, -1, :].view(batch_size, self.num_entities, -1)
```

### 改成

```python
z = self.temporal_encoder(z)
z_last = z[:, -1, :]
z_mean = z.mean(dim=1)
z = 0.7 * z_last + 0.3 * z_mean
z = z.view(batch_size, self.num_entities, -1)
```

这对短时交通预测很合理，因为 12 分钟历史里的趋势比最后 1 分钟更稳。

---

## Patch D：V2 单独训练超参数

当前 V2 和 V1 共享 `lr=1e-3, epochs=12, patience=4`，这个对 V2 太短。([GitHub][3])

建议改为：

```python
if kind == "transformer_v2":
    lr = 3e-4
    weight_decay = 5e-5
    max_epochs = 40
    patience = 8
    grad_clip = 1.0
else:
    lr = 1e-3
    weight_decay = 1e-4
    max_epochs = 12
    patience = 4
    grad_clip = None
```

如果担心训练慢，可以先：

```bash
set TRAFFIC_TRAIN_EPOCHS_V2=25
set TRAFFIC_TRAIN_PATIENCE_V2=6
```

---

## Patch E：V2 模型尺寸小幅上调

当前：

```python
d_model=96
temporal_layers=2
spatial_layers=2
dropout=0.1
```

建议第一轮改成：

```python
d_model=128
temporal_layers=2
spatial_layers=1
dropout=0.05
```

理由：

* `d_model=96` 比 V1 的 128 弱；
* spatial_layers=2 在无 graph bias 情况下容易让 151 个 movement 全连接噪声互相污染；
* graph residual 后，spatial encoder 可以先保持 1 层；
* dropout 0.1 对小数据 + 共享 head 的 V2 可能偏大。

# 8. 我对 V2 不如 V1 的最终判断

按优先级排序，原因应该是：

## 第一原因：V2 没用 graph，空间编码是无约束全连接 attention

你们已经有 `movement_graph.json`，且关系类型非常贴合信号控制场景，但 V2 当前完全没用。
这会让 151 个 movement 之间无差别互相关注，空间 attention 的噪声会很大。

## 第二原因：V2 的共享 head 过强，entity-specific 输出能力不足

V1 直接输出 `15 × 453`，每个 movement/target/horizon 都有独立输出权重；V2 每个 movement 共用一个 `15 × 3` head，只靠 entity embedding 区分 movement。对你们这种 151 个 movement 的局部路网，V2 很可能被限制了表达力。

## 第三原因：temporal-first + last-only pooling 丢失了跨 movement 的时序交互

当前 spatial encoder 只能看到每个 movement 的最后 temporal summary，而不是完整 12 步历史里的空间关系。([GitHub][2])

## 第四原因：V2 的训练预算偏小

`12 epochs + patience 4 + lr 1e-3` 对 V1 可以，但对 V2 的双 encoder、graph residual、movement embedding 类结构不够友好。([GitHub][3])

## 第五原因：phase_id 被当作连续变量

`phase_id` 是类别变量，但现在只是普通数值输入。短期可以先不改；中期建议做 phase embedding 或将相位改成 one-hot/绿灯状态特征。

# 9. 最小实验顺序

不要一次性全改，否则不知道哪个改动有效。建议按这个顺序跑：

```text
Exp-0: 当前 V2 baseline
Exp-1: V2 + entity_output_bias
Exp-2: Exp-1 + last_mean_pooling
Exp-3: Exp-2 + V2 专属超参数
Exp-4: Exp-3 + graph_residual
Exp-5: Exp-4 + d_model=128, spatial_layers=1, dropout=0.05
```

最关键的对照指标不要只看 overall MAE，还要看：

```text
overall
incident
control_perturbation
h5 / h10 / h15
arrival_flow / mean_speed / queue_veh 分目标指标
turn_type = l / s / r 分转向指标
tls_id 分路口指标
```

当前 `movement_catalog_quality.json` 已经说明 movement catalog 本身质量是可用的：151 个 movement，14 个 tls，重复 ID 为 0，missing phase 为 0，zone quality 全部为 ok。
所以现在的重点不是再修 catalog，而是让 V2 真正利用 movement 拓扑和相位结构。
